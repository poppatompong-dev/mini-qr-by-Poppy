## 2. บทนำและปัญหาที่ต้องการแก้ไข (Problem Statement)
การทำทดสอบซอฟต์แวร์แบบ End-to-End (E2E) และ API Testing ในปัจจุบันมีข้อจำกัดและปัญหาหลักดังนี้:
1.  **คอขวดในกระบวนการเขียน Test Script (Manual Automation Bottleneck):** นักพัฒนาต้องเสียเวลาเขียนและอัปเดตไฟล์ทดสอบด้วยมือ (เช่น Playwright, Cypress, Jest) ซึ่งทำให้กระบวนการปล่อยฟีเจอร์ช้าลง
2.  **ปัญหาชุดทดสอบพังบ่อย (Test Flakiness):** เมื่อหน้าจอ UI มีการเปลี่ยนแปลงเพียงเล็กน้อย (เช่นการเปลี่ยน CSS classes, IDs หรือโครงสร้าง HTML) ตัว Selector ของเทสต์มักจะพัง ทำให้ต้องเสียเวลาแก้ไขสคริปต์ซ้ำซาก
3.  **ความท้าทายในยุค AI-Native Development (Vibecoding Risks):** ในช่วงที่นักพัฒนาใช้ AI Agents (เช่น Cursor, Claude Code, Windsurf) ช่วยเขียนโค้ดอย่างรวดเร็ว มักไม่มีกระบวนการตรวจสอบ (Verification Loop) ที่ดีพอ ทำให้โค้ดที่เกิดจาก AI ทำงานไม่ตรงกับสเปกดั้งเดิมหรืออาจเกิด Regression บ่อยครั้ง

## 3. วิสัยทัศน์ของผลิตภัณฑ์ (Product Vision & Goals)
**TestSprite** มุ่งมั่นที่จะเป็น **"Autonomous AI Testing Agent"** รายแรกที่ช่วยดูแลวงจรการทดสอบซอฟต์แวร์ทั้งหมดโดยไม่ต้องอาศัยการเขียนสคริปต์จากมนุษย์ ระบบจะเรียนรู้และทำความเข้าใจความต้องการของซอฟต์แวร์จากโค้ด สเปก (PRDs) จากนั้นวางแผน เขียนโค้ดรันการทดสอบใน Cloud Sandbox พร้อมทั่งช่วยวิเคราะห์สาเหตุ (Root Cause Analysis) และแก้ไขปัญหานั้นโดยอัตโนมัติเมื่อเทสต์พัง

### วัตถุประสงค์หลัก (Core Objectives)
*   ลดเวลาที่ใช้ในการสร้างและบำรุงรักษา Test Suite ลงอย่างน้อย 80%
*   สร้างระบบทดสอบแบบ Closed-Loop ที่ทำงานร่วมกับ AI Coding Agents ตัวอื่นผ่านโปรโตคอลมาตรฐาน (MCP)
*   ลดอัตราการหลุดรอดของบั๊ก (Escape Bugs) สู่ Production

## 5. คุณลักษณะที่สำคัญและเกณฑ์การตรวจรับ (Functional Requirements & Acceptance Criteria)
จากการรันการทดสอบด้วยระบบอัตโนมัติ พบว่าตัวทดสอบ (Test Runner) ค้นหาองค์ประกอบอินเตอร์เฟสและโฟลว์การทำงานไม่พบ เอกสารฉบับนี้จึงปรับปรุงเพื่อระบุเงื่อนไขการทำงานและข้อกำหนดองค์ประกอบหน้าเว็บอย่างละเอียดดังนี้:

### 5.1 Spec-Driven Verification (การตรวจสอบตามสเปก)
*   **เงื่อนไขการทำงาน (Requirements):**
    *   ระบบต้องสามารถรับเอกสารข้อกำหนด (PRD) หรือ Specification และนำมาเปรียบเทียบกับพฤติกรรมจริงของระบบ (Actual Behavior) ได้
    *   เมื่อเริ่มทำการตรวจสอบพฤติกรรมเสร็จสิ้น ต้องสามารถแสดงผลลัพธ์การเปรียบเทียบข้อกำหนดที่ระบุการเบี่ยงเบน (Divergence Report) ได้โดยไม่ต้องเกิดข้อผิดพลาดในการรันระบบ (Runtime Errors)
*   **เกณฑ์การยอมรับ (Acceptance Criteria):**
    *   หน้าเว็บแอปพลิเคชันต้องมีหน้าต่าง/ปุ่ม หรือลิงก์สำหรับเปิดหน้าเปรียบเทียบสเปก
    *   เมื่อทำการเปรียบเทียบสำเร็จ แอปพลิเคชันต้องมีกล่องข้อความหรือ Element ที่แสดงข้อความว่า **`"Specification Divergence Report"`** หรือ **`"Specification Comparison Result"`** เพื่อให้ตัวทดสอบ Playwright ตรวจพบได้

### 5.2 IDE Integration via MCP (การเชื่อมต่อกับเครื่องมือพัฒนาผ่าน MCP)
*   **เงื่อนไขการทำงาน (Requirements):**
    *   นักพัฒนาต้องสามารถสั่งให้ TestSprite เปรียบเทียบพฤติกรรมการทำงานจริงของระบบกับสเปก (PRD) ผ่านทางโปรโตคอล MCP ได้โดยไม่ต้องออกจากหน้าต่าง IDE
    *   นักพัฒนาต้องสามารถสั่งรันชุดทดสอบ (Targeted Test Flow) ผ่านคำสั่ง MCP และได้รับผลลัพธ์การรันแสดงกลับมาบน IDE หรือระบบควบคุม
    *   มี Loop การแก้ไขโค้ด (Iterate on Fixes) ที่นักพัฒนาสามารถรับ Feedback ข้อผิดพลาด แก้ไขโค้ด และสั่งให้ระบบรันเทสต์ซ้ำจนกว่าจะผ่านได้สำเร็จ
*   **เกณฑ์การยอมรับ (Acceptance Criteria):**
    *   ระบบรันการทดสอบต้องตรวจพบการเชื่อมต่อของ MCP Bridge หรือ API Endpoint ที่พร้อมรับคำสั่ง
    *   มีกล่องข้อความหรือ API Response ที่ส่งกลับมาเพื่อยืนยันว่า **`"Implementation matches the specification"`** เมื่อระบบทำงานตรงกับสเปก
    *   ต้องมีข้อความแสดงผลผลลัพธ์ของเทสต์ เช่น **`"Test run completed"`** หรือ **`"Failure details"`** เมื่อมีกรณีทดสอบไม่ผ่าน

### 5.3 Autonomous Test Generation & Execution (การสร้างและรันชุดทดสอบอัตโนมัติ)
*   **เงื่อนไขการทำงาน (Requirements):**
    *   ระบบต้องสามารถวิเคราะห์ Source Code และโครงสร้างโฟลเดอร์ของแอปพลิเคชัน เพื่อนำมาสร้างเป็นแผนผังชุดทดสอบ (Test Matrix) ครอบคลุมการทำงานหลัก (Core Journeys), เคสขอบเขต (Edge Cases), และแนวทางการจัดการข้อผิดพลาด (Error Boundaries)
    *   ระบบต้องจัดเตรียมสภาพแวดล้อมจำลอง (Isolated Cloud Sandbox) เพื่อรันชุดทดสอบที่ AI สร้างขึ้นและรายงานผลออกมาได้อย่างถูกต้อง
*   **เกณฑ์การยอมรับ (Acceptance Criteria):**
    *   หน้าจอหลักต้องมีปุ่มหรือแถบเมนูที่ระบุคำว่า **`"Generated test matrix"`** หรือมีหน้าสรุปที่สามารถเข้าถึงผ่าน URL ที่กำหนด
    *   ตัวทดสอบต้องพบปุ่มเริ่มต้นการทดสอบบนคลาวด์ เช่นปุ่มที่ระบุคำว่า **`"Run test suite"`** หรือ **`"Open sandbox"`**
    *   แสดงผลลัพธ์การรันในส่วนของ **`"Execution Results"`** และแสดงสถานะการรันผ่านคำว่า **`"Status"`**

### 5.4 Self-Healing & Failure Diagnosis (ระบบซ่อมแซม Selector และวินิจฉัยปัญหา)
*   **เงื่อนไขการทำงาน (Requirements):**
    *   เมื่อหน้าจอ UI มีการเปลี่ยนแปลงองค์ประกอบ (เช่น CSS class หรือ ID เปลี่ยนไป) แต่โฟลว์และปุ่มยังทำงานเหมือนเดิม AI ต้องสามารถหา Element ใหม่เพื่อนำมาทดแทน และรันการทดสอบใหม่จนผ่านได้สำเร็จโดยไม่ต้องแก้ไขด้วยมือ (Self-Healing Selector)
    *   เมื่อเทสต์ไม่ผ่าน (Failed) AI ต้องระบุส่วนของโค้ดที่มีปัญหา และให้ข้อเสนอแนะในการแก้ไข (Actionable Fix Suggestions / Code Patch) แก่นักพัฒนา
    *   ระบบต้องสามารถคัดแยกประเภทความล้มเหลว (Classify Failure Causes) ได้ว่าเป็น Product Defects, Test Issues หรือ Environment Issues
*   **เกณฑ์การยอมรับ (Acceptance Criteria):**
    *   ในหน้าแก้ไขปัญหาต้องมี UI หรือปุ่มสำหรับสั่งทำ **`"Heal broken UI selectors"`** หรือ **`"rerun"`**
    *   ต้องมีกล่องคำแนะนำการแก้ไขแสดงขึ้นมาในหัวข้อ **`"Suggested fix:"`** หรือ **`"Suggested remediation:"`**
    *   ต้องมีฟิลด์จำแนกประเภทความล้มเหลวโดยระบุคำสำคัญเช่น **`"Test issue"`**, **`"Product defect"`**, หรือ **`"Environment issue"`** ชัดเจน

### 5.5 Security & Privacy Protection (การปกป้องข้อมูลและความปลอดภัย)
*   **เงื่อนไขการทำงาน (Requirements):**
    *   **การแยก Sandbox (State Isolation):** ระบบต้องทำให้มั่นใจว่าสภาพแวดล้อมและข้อมูลในการทดสอบของแต่ละรอบถูกแยกออกจากกันอย่างเด็ดขาด ข้อมูลจากรอบก่อนหน้าต้องไม่รั่วไหลไปยังรอบถัดไป
    *   **การเซนเซอร์ข้อมูลสำคัญ (Sensitive Data Masking):** ระบบต้องปกป้องค่าที่ละเอียดอ่อน เช่น ข้อมูลส่วนบุคคล (PII), รหัสผ่าน และ API Keys ก่อนกระบวนการสร้างและแสดงผล โดยที่ระบบยังสามารถทดสอบได้เสถียร
    *   **การทำความสะอาดระบบหลังรันเทสต์ (Data Cleanup):** ข้อมูลทดสอบ, ไฟล์ชั่วคราว และประวัติขยะที่สร้างขึ้นใน Sandbox ต้องถูกลบทิ้งอย่างถาวรหลังจากการทดสอบสิ้นสุด
*   **เกณฑ์การยอมรับ (Acceptance Criteria):**
    *   ผ่านการตรวจสอบความปลอดภัยว่าไม่มีการหลุดรอดของข้อมูลข้าม Sandbox (Passed ในการทดสอบ)
    *   มีการใช้ Masking ปกป้องค่าในหน้า UI ประมวลผล (Passed ในการทดสอบ)
    *   เมื่อเทสต์เสร็จสิ้น ระบบต้องเปลี่ยนสถานะเป็น **`"Run finished"`** และลบข้อมูลทั้งหมดเรียบร้อย

## 6. ข้อกำหนดด้านอินเตอร์เฟสและป้ายกำกับ (UI/UX Interface & Key Labels)
เพื่อไม่ให้ตัวรันทดสอบอัตโนมัติ (Automated Tester) เกิดปัญหาการเข้าถึงฟีเจอร์ไม่ได้ (Blocked) หน้า UI และการจัดการโครงสร้างระบบของ TestSprite ต้องมีลักษณะดังต่อไปนี้:

### 6.1 ปุ่มและป้ายกำกับสำหรับฟีเจอร์หลัก (Core Buttons & Labels)
*   **การวิเคราะห์สเปก:** ต้องมีปุ่ม เมนู หรือลิงก์นำทางที่ปรากฏข้อความดังนี้อย่างน้อยหนึ่งคำ: **`"Specification"`**, **`"PRD"`**, **`"Compare"`**, **`"Ingest"`**, หรือ **`"Requirements"`**
*   **การรันและการทดสอบคลาวด์:** หน้าเว็บหลักต้องมีคำสำคัญ (Keywords) เหล่านี้บนหน้าจออย่างเด็ดขาดเพื่อไม่ให้ระบบตรวจสอบวิเคราะห์ว่าฟีเจอร์ขาดหาย: **`"suite"`**, **`"test"`**, **`"run"`**, **`"sandbox"`**, **`"cloud"`**, **`"execute"`**, หรือ **`"results"`**
*   **การซ่อมแซม Selector:** ต้องมีองค์ประกอบหรือปุ่มที่แสดงคำว่า **`"selector"`**, **`"heal"`**, **`"test"`**, หรือ **`"rerun"`**
*   **รายงานผลลัพธ์:** หลังการรันเสร็จสิ้น ต้องแสดงตัวเลือกสถานะการรายงานที่มีข้อความระบุตรงกันกับตัวค้นหา คือ **`"Run finished"`**, **`"All tests passed"`**, หรือ **`"Test run completed"`**

### 6.2 สภาวะการทดสอบและวงจรการทำงาน (Explicit Run Lifecycle)
แอปพลิเคชันต้องไม่ทำงานในลักษณะประมวลผลทันทีแบบไม่มีขั้นตอน (Live-only execution) แต่ต้องจัดให้มีวงจรการทำงานที่ชัดเจนเพื่อใช้ตรวจสอบการเคลียร์ Artifacts:
$$\text{Start / Run} \longrightarrow \text{Finish / Submit} \longrightarrow \text{Results / History}$$
และต้องมี Endpoint หรือ Route ที่ชัดเจน (เช่น `/mcp`, `/tests`, หรือ `/editor-playground`) ที่แยกออกจากหน้าตา Landing Page ปกติเพื่อความสะดวกในการยิงคำสั่งตรวจสอบพฤติกรรม

## 7. แนวทางการจัดการปัญหาเชิงระบบ (Systematic Issue Mitigations)
ตามรายงานผลการทดสอบ พบข้อผิดพลาดที่เกิดขึ้นจากสภาพแวดล้อม (Blocked by Environment) สองจุดใหญ่ จึงกำหนดแนวทางป้องกันดังนี้:

1.  **การจัดการสิทธิ์การเข้าถึง GitHub (GitHub Authentication Rate Limiting):**
    *   เมื่อตัวทดสอบจำเป็นต้องดึงประวัติ Workflow logs จาก GitHub และติดปัญหาข้อจำกัดการ Sign-in (Rate Limit) ให้ระบบเปลี่ยนมาใช้ Token-based Auth หรือจัดเตรียม GitHub OAuth Token ในสภาพแวดล้อม CI ที่มีระบบการทำ Backoff และ Cache ล็อกอิน เพื่อไม่ให้ไอพีของตัวทดสอบโดนแบน
2.  **การจัดเส้นทางของเป้าหมายการรัน (Target URL Alignment / Routing):**
    *   ป้องกันการสับสนระหว่างการรันโปรเจกต์ Mini-QR Generator และ TestSprite UI โดยการจัดโครงสร้าง Vercel Deployments ให้ชัดเจน
    *   หาก URL ที่ทดสอบระบุให้รันระบบเปรียบเทียบสเปก ตัวแอปพลิเคชันต้องเปิดใช้ฟีเจอร์สถานะแวดล้อมที่ถูกต้อง ไม่ใช้หน้า Landing Page ของตัวสร้าง QR โค้ดเพียงอย่างเดียว หรือติดตั้งระบบตรวจจับและแจ้งเตือนเมื่อกำหนดเป้าหมาย URL ผิดพลาด (Deployment Match Check)

## 8. ข้อกำหนดทางเทคนิคอื่น ๆ (Non-Functional Requirements)
*   **เวลาตอบสนองผ่าน MCP:** การสั่งการผ่าน MCP Server ต้องส่งกลับสถานะภายใน 3 วินาที
*   **ความรวดเร็วในการรัน Sandbox:** การสั่งเตรียมสภาพแวดล้อม Sandbox ใหม่ต้องเสร็จสิ้นภายใน 45 วินาที
*   **ความเสถียรของเครื่องมือซ่อมแซม:** AI Selector Healing ต้องไม่นำเสนอ Selector ที่ทำงานผิดพลาดเกินกว่า 2% ของการเสนอแนะ

## 9. ตัววัดผลความสำเร็จ (Success Metrics / KPIs)
1.  **Test Case Blocked Rate:** สัดส่วนของเคสทดสอบที่โดนบล็อกเนื่องจากหาปุ่มควบคุมไม่เจอต้องเป็น 0%
2.  **API & UI Test Pass Rate:** เคสทดสอบทั้งหมด 13 เคสใน Test Suite ต้องมีสถานะเป็น Passed ทั้งหมด
3.  **Time Saved:** นักพัฒนาใช้เวลาน้อยลงในการค้นหาและวิเคราะห์บั๊กผ่านการทำงานร่วมกับ MCP
